## 📱 What Is Assignment 5

Assignment 5 extends the National Parks Explorer (Assignment 4) with **Firebase Authentication**. Users must sign up or log in before accessing the app. The authentication layer is built as a clean separation from the existing NPS API functionality — all new auth files are isolated in their own models, services, providers, screens, and widgets.

### ✅ Criterion 3 — User Session Management (25 pts)

| Requirement | Implementation |
|---|---|
| Responds on login | `Navigator.pushAndRemoveUntil → IntroScreen` |
| Responds on logout | `signOut()` → `Navigator.pushAndRemoveUntil → LoginScreen` |
| Correct screen based on auth state | `AuthGate` uses `StreamBuilder<User?>` on app launch |
| Prevents access when not logged in | `AuthGate` always shows `LoginScreen` if no Firebase user |
| Persistent session | Firebase persists the session — reopening app skips login |

**AuthGate implementation:**
```dart
StreamBuilder<User?>(
stream: FirebaseAuth.instance.authStateChanges(),
builder: (context, snapshot) {
if (snapshot.connectionState == ConnectionState.waiting)
return const Scaffold(body: Center(child: CircularProgressIndicator()));
if (snapshot.hasData && snapshot.data != null)
return const IntroScreen();   // logged in
return const LoginScreen();     // not logged in
},
)
```

### ✅ Criterion 4 — Error Handling & Code Organization (25 pts)

| Requirement | Implementation |
|---|---|
| Empty field validation | Client-side `FormField` validators before any Firebase call |
| Invalid credentials | `invalid-credential` → "Incorrect email or password" |
| Failed sign up | `email-already-in-use`, `weak-password` → friendly messages |
| 10 Firebase error codes mapped | `_friendlyError()` in `AuthService` |
| Code organized into layers | `models/` · `services/` · `providers/` · `screens/` · `widgets/` |
| Reusable components | `AuthFormField` widget used across Login, Signup, ForgotPassword |

**Error code mapping:**
```dart
String _friendlyError(String code) => switch (code) {
  'email-already-in-use'   => 'An account with this email already exists.',
  'invalid-email'          => 'Please enter a valid email address.',
  'weak-password'          => 'Password must be at least 6 characters.',
  'user-not-found'         => 'No account found with this email address.',
  'wrong-password'         => 'Incorrect password. Please try again.',
  'invalid-credential'     => 'Incorrect email or password. Please check and try again.',
  'user-disabled'          => 'This account has been disabled.',
  'too-many-requests'      => 'Too many attempts. Please try again later.',
  'network-request-failed' => 'Network error. Check your connection.',
  'channel-error'          => 'Please fill in all fields.',
  _                        => 'Authentication failed (code: $code). Please try again.',
};
```

## 🔑 Key Design Decisions

**`AuthGate` is launch-only.** It uses `StreamBuilder<User?>` to decide the first screen when the app cold-starts. After that, every screen manages its own navigation explicitly using `Navigator.pushAndRemoveUntil` — this avoids the "AuthGate rebuilds underneath the nav stack" problem where the user never sees the state change.

**Auto sign-in after signup.** Firebase automatically signs users in after `createUserWithEmailAndPassword`. The app embraces this — after the 2-second success banner, users land directly in the app without needing to log in again.

**`AuthUser` wrapper.** Rather than passing `firebase_auth.User` throughout the app, a lightweight `AuthUser` model is used. It provides a `greeting` getter that returns `displayName`, or the email prefix, or `"Explorer"` as fallback — so the AppBar always has something sensible to show.

**Validation is two-layer.** Client-side `FormField` validators run first (empty check, email format, password length, confirm match) before any Firebase call is made. Firebase error codes handle the server-side failures.

## 📱 Screen Guide

### Login Screen
- Email + password fields with inline validation
- Password visibility toggle
- Red error banner for wrong credentials
- "Forgot password?" link → ForgotPasswordScreen
- "Sign Up" link → SignUpScreen

### Sign Up Screen
- Full name + email + password + confirm password
- Confirm password must match (validated client-side)
- Green success banner on account creation: "Account created! Welcome, [name]."
- Button changes to "Account Created!" and disables
- Navigates to IntroScreen after 2 seconds

### Forgot Password Screen
- Email input with format validation
- "Send Reset Link" button with loading state
- Always shows success view after valid email submit (security — doesn't reveal if email exists)
- Success message: "If an account exists for [email], you will receive a reset link shortly."

### Intro Screen (updated)
- AppBar shows user's initial avatar + greeting name + email
- Red logout icon (top-right) → confirmation dialog → signs out → LoginScreen
